# Flow-Version Workspace UI Rework

## Summary

This is the direction doc for the ongoing rework of `apps/web-app` away from a page-navigation web app toward a flow-version-centric workspace: drill into one flow version, and work with it entirely through panels/tabs for whichever mode you're in (Edit, View, Run, Run History, Sims, Artifacts, Evals) — rather than navigating to separate top-level pages for runs/sims/artifacts/evals that all have to be manually cross-referenced by id.

Not everything below is equally fleshed out. The point of writing it now is to capture the parts that *are* settled (so they don't need re-deriving later) and name the parts that aren't (so they read as open, not forgotten). This doesn't map to a single GitHub Milestone or version bump — each concrete chunk below (starting with Run History) is its own small, closeable milestone; this doc is the throughline connecting them.

## Overall idea and purpose

The core problem with the current page-based UI: nearly everything lives at the "list all runs / list all sims / list all artifacts" level, globally, with no real organizing entity underneath it. That's the wrong default — the actual daily unit of work is *one specific flow version*. The rework makes that the primary workspace:

- **Flow version = the primary workspace.** Drill into one, and within that single context: edit it (if still a draft), run it, view its run history, create/view sims of it, browse/create artifacts scoped to it, construct and run evals against it — all via panels/tabs inside one persistent shell, VS-code-style (frames with tabs, panels loading different content).
- **Flow (all versions) = one level up.** Comparing sims/runs/artifacts/evals across every version of one flow — needed, but not the primary workspace.
- **Global (all flows/all runs/etc.) = top, rare.** Useful sometimes as a filterable escape hatch, but explicitly not where work usually happens — today's UI is almost entirely this level, which is the concrete reason it's been "not very useful."

Longer-term context worth keeping in view even though it's not being built yet: the eventual target runtime is Tauri (not just a browser tab), with real multi-pane coexistence (view + edit + preview at once, like an IDE) rather than one panel at a time, and an explicit non-goal of supporting small/phone-sized screens — the panel/toolbar model is inherently a desktop pattern. Also an explicit constraint: this UI is being hand-designed and understood, not generated wholesale by an AI/design tool in one shot, the same ownership stance as the rest of this codebase.

## What's already built

- The routing shell: `/spike/:flowId/:versionId/<mode>` via `FlowVersionWorkspace` (real flow-version fetch) + `FlowVersionModeNav`, replacing the old flat `Spike.tsx` prototype. One persistent shell, one `Outlet` per mode, state scoped per mode rather than globally.
- **Edit / View modes**: routed and rendering (flow graph + JSON view, step details, settings/params/problems panels), but not yet functionally distinct from each other — Edit isn't actually editable yet. Matches the concrete panel pairing already envisioned: flow graph + JSON definition for authoring context.
- **Run mode**: fully working end to end — submit a run, watch it live (websocket-fed events, no polling), an animated flow chart showing live step status (pending/running/completed/failed) derived client-side from the run's own event stream, a params panel with artifact selection, and a preview feature showing how a selected param actually resolves everywhere it's referenced in the flow (including refs embedded in larger template strings). Matches the other envisioned panel pairing: flow graph + step execution detail for run/inspection context.
- **Step Results** (Run mode's right panel, PR #261): status, Output & Exports, Field Resolution (a field's final value with every `{{...}}` ref in it substituted, composed across multiple refs in one field), and References (each individual reference's own resolved value) — all sourced from the run's own event stream, deliberately with zero dependency on `job.<cap>.submitted`'s payload shape (see `docs/todo.md`).
- Reusable pieces this unlocked for later modes: `EventGraph` (decoupled from any one page's state, takes events/selection as plain props), `deriveStepRunInfo`/`useStepRunInfo` (a pure fold over any event array, not tied to live-only data — status plus output/export hashes), `findStepRefs`/`resolveRefHash`/`foldResolvedField` (`lib/ref-resolution.ts`, also event-array-agnostic).

## Next bit of work: Run History

The next scoped, closeable chunk (Step Output shipped as part of Run mode's Step Results, above). Planned as a few small PRs rather than one, in this order:

1. **Backend: a flow-version-filtered run list endpoint/query.** `useListAllRunsQuery` has no flow-version filter today — small, testable on its own, no UI dependency.
2. **Run History scaffold with real data, no visualization yet.** Left-panel run list (using the new filtered query) wired to routing; selecting a run loads its full historical event list via `useGetAllRunEventsQuery` (the same REST backfill Run mode already uses for live backfill) — proves the pipe works before anything renders it.
3. **Wire in Run mode's existing pieces** (`FlowGraph`+step statuses, `EventGraph`, `StepResultsTab`) fed by the selected historical run's events instead of the live one. This is where the container-level "shared vs. Run-mode-specific" question for `FlowVersionRunGraphPanel`/`FlowVersionRunDetailsPanel` gets a real answer, with a second actual consumer in hand instead of a guess.

## Other bits of work (less fleshed out — sizes and shapes still provisional)

- **Artifacts mode** — browse/create artifacts scoped to a specific flow/flow-version, rather than the current global artifact list. Real backend piece (some association between artifacts and a flow/flow-version entity) plus a UI tab. Estimated smaller than the Evals rework, but not sized precisely yet.
- **Sims mode** — likely reuses existing `RunnerSimSelector`/`Sims.tsx`/`CreateSim.tsx`/`ViewSim.tsx` pieces, scoped to the current flow version, the same way Run mode reused `RunnerParamRow`. Not yet looked at closely enough to size confidently.
- **Edit mode, actually editable** — currently just a read-only copy of View. Real scope unknown; likely a substantial lift given flow authoring/validation/publish-state semantics aren't designed yet (e.g. whether a published version becomes immutable and editing forks a new draft version).
- **A full-step "resolved preview" view — parked, not scoped.** Raised while building Step Results: a view showing *every* field of a step (matching `StepHttpJsonDetails`' full field list — URL/Method/Headers/Body/Exports — not just fields with refs), with each field showing its bound param value before a run and its actual resolved value after. Two distinct gaps motivate this, not one:
  - Fields with zero refs are invisible in Run mode entirely — only Edit/View's `StepHttpJsonDetails` shows them, and only unresolved.
  - Fields that *do* have refs are only ever shown at leaf-`bindPath` granularity in Field Resolution (e.g. just `body.messages[1].content`), never composed back up into the whole logical field (`body`, `headers`) as it was actually sent — if `body` has refs at two different nested paths, Field Resolution shows two disjoint leaf rows, never one composed `body` object with both substitutions sitting in their natural positions. The natural grain for "the whole field, fully resolved" is the same field-level granularity `StepHttpJsonDetails` already uses; Field Resolution deliberately went finer because "what did this one reference become" wants the leaf, not the whole object.

  Also explicitly not Run-History-specific, and connects to an existing feature, not just Field Resolution: the per-param "show usages" report (`findParamRefs`/`renderParamRefReport`, `FlowVersionRunParamRow`) already lets you click a *param* and see a markdown report of everywhere it's used — useful, but it's the reverse direction (param → its usages, leaf-level, textual report). What's missing is the other direction: click a *step* in the live Params panel and see that whole step's fields composed with the currently-selected param artifacts filled in, ahead of submitting — same underlying resolution logic (`resolveRefHash`/`buildRefUsage`/`foldResolvedField`), organized by step instead of by param, at whole-field instead of leaf granularity. Deliberately not built yet because it's unclear how much it'd repeat Field Resolution/References; the resolution reached in discussion: judge that by whether it answers a *different question* (a holistic "what did this step look like as one document" snapshot vs. drill-down provenance), not by whether it reuses the same underlying resolved-value logic — reusing `foldResolvedField` etc. across views isn't repetition, showing the same field list with the same framing twice would be. Needs its own design pass before it's scope, the same way Step Results got one.
- **Replay — acknowledged valuable, not scheduled (not "no reason exists" anymore).** Watching a historical run's execution animate at some controlled pace (a play-head over the event timeline instead of a range selection, reconstructing state at any point rather than just the final one) — technically straightforward given the architecture (`deriveStepRunInfo` already works on any event array, live or historical; this is UI/timing work, not new plumbing). The concrete reason it's worth building, not just "it'd be cool": it lets you re-experience the feel of watching a run live — pacing, what happened when — which a final-state view structurally can't reproduce. Still not the first thing to build; no committed timing.

## Evals: deliberately its own separate, future milestone

Explicitly not part of this arc's near-term scope — this is a real data-model rearchitecture, not a UI tab, and deserves its own milestone the same way the original eval vertical slice did (Milestone #9). Current state: evals are embedded inside the flow definition itself (an eval is just a flow with a judge step) — a smaller move than originally wanted. The actual goal: evals as standalone, reusable entities, decoupled from any one flow — a fixed template (System Prompt + Question + Context + Answer) that can be constructed and run against different flows. Inside this workspace, an Evals mode/tab would let you construct an eval run (bind the flow's actual output to the eval template's expected inputs) — results visualization might end up needing its own separate page if it doesn't fit inside that tab. Not scoped further here; needs its own dedicated design pass.

## Process note

Milestones here don't need to map 1:1 to version bumps or a formal GitHub Milestone object — each chunk above can close independently, with version bumps happening at whatever later point represents a real, demo-able jump (e.g. "the revamped UI once the pages are mostly done"), not tied to milestone count.
