# Flow-Version Workspace UI Rework

## Summary

This is the direction doc for the ongoing rework of `apps/web-app` away from a page-navigation web app toward a flow-version-centric workspace: drill into one flow version, and work with it entirely through panels/tabs for whichever mode you're in (Edit, View, Run, Run History, Sims, Artifacts, Evals) — rather than navigating to separate top-level pages for runs/sims/artifacts/evals that all have to be manually cross-referenced by id.

Not everything below is equally fleshed out. The point of writing it now is to capture the parts that _are_ settled (so they don't need re-deriving later) and name the parts that aren't (so they read as open, not forgotten). This doesn't map to a single GitHub Milestone or version bump — each concrete chunk below (starting with Run History) is its own small, closeable milestone; this doc is the throughline connecting them.

## Overall idea and purpose

The core problem with the current page-based UI: nearly everything lives at the "list all runs / list all sims / list all artifacts" level, globally, with no real organizing entity underneath it. That's the wrong default — the actual daily unit of work is _one specific flow version_. The rework makes that the primary workspace:

- **Flow version = the primary workspace.** Drill into one, and within that single context: edit it (if still a draft), run it, view its run history, create/view sims of it, browse/create artifacts scoped to it, construct and run evals against it — all via panels/tabs inside one persistent shell, VS-code-style (frames with tabs, panels loading different content).
- **Flow (all versions) = one level up.** Comparing sims/runs/artifacts/evals across every version of one flow — needed, but not the primary workspace.
- **Global (all flows/all runs/etc.) = top, rare.** Useful sometimes as a filterable escape hatch, but explicitly not where work usually happens — today's UI is almost entirely this level, which is the concrete reason it's been "not very useful."

Longer-term context worth keeping in view even though it's not being built yet: the eventual target runtime is Tauri (not just a browser tab), with real multi-pane coexistence (view + edit + preview at once, like an IDE) rather than one panel at a time, and an explicit non-goal of supporting small/phone-sized screens — the panel/toolbar model is inherently a desktop pattern. Also an explicit constraint: this UI is being hand-designed and understood, not generated wholesale by an AI/design tool in one shot, the same ownership stance as the rest of this codebase.

## What's already built

- The routing shell: `/spike/:flowId/:versionId/<mode>` via `FlowVersionWorkspace` (real flow-version fetch) + `FlowVersionModeNav`, replacing the old flat `Spike.tsx` prototype. One persistent shell, one `Outlet` per mode, state scoped per mode rather than globally.
- **Edit / View modes**: routed and rendering (flow graph + JSON view, step details, settings/params/problems panels), but not yet functionally distinct from each other — Edit isn't actually editable yet. Matches the concrete panel pairing already envisioned: flow graph + JSON definition for authoring context.
- **Run mode**: fully working end to end — submit a run, watch it live (websocket-fed events, no polling), an animated flow chart showing live step status (pending/running/completed/failed) derived client-side from the run's own event stream, a params panel with artifact selection, and a preview feature showing how a selected param actually resolves everywhere it's referenced in the flow (including refs embedded in larger template strings). Matches the other envisioned panel pairing: flow graph + step execution detail for run/inspection context.
- **Step Results** (Run mode's right panel, PR #261): status, Output & Exports, Field Resolution (a field's final value with every `{{...}}` ref in it substituted, composed across multiple refs in one field), and References (each individual reference's own resolved value) — all sourced from the run's own event stream, deliberately with zero dependency on `job.<cap>.submitted`'s payload shape (see `docs/todo.md`).
- Reusable pieces this unlocked for later modes: `EventGraph` (decoupled from any one page's state, takes events/selection as plain props), `deriveStepRunInfo`/`useStepRunInfo` (a pure fold over any event array, not tied to live-only data — status plus output/export hashes), `findStepRefs`/`resolveRefHash`/`foldResolvedField` (`lib/ref-resolution.ts`, also event-array-agnostic).
- **Run History mode**: fully built across 3 small PRs (#262, #263, #264) — a flow-version-filtered run list (`listRunsByFlowVersionId`), a left-panel run list wired to routing with REST event backfill on selection, and Run mode's own graph/details panels (`FlowVersionRunGraphPanel`/`FlowVersionRunDetailsPanel`, `StepResultsTab`) reused as-is, fed by the selected historical run's events instead of the live one — confirmed those panels needed zero changes, since they were already pure props-in/callbacks-out with no coupling to the live Redux slice. Also produced two small shared extractions used by both Run and Run History: `useRunEventsWithStatus` (events backfill + `deriveStepRunInfo`, identical logic regardless of live vs. historical) and `lib/run-panel-state.types.ts` (tab/focused-content types hoisted out of the live slice, since they were never actually live-specific — each mode still keeps its own independent slice state, no shared/merged state between modes).
- **Sims mode**: fully built across 5 PRs (#265, #266, #267, #268, #269) — see "Sims mode" section below for the full design narrative (state machine, placement decisions, corrections). Summary: authoring a new sim reuses Run History's graph/details pieces with an added reused-step toggle; browsing _is_ viewing (the sim list stays visible and freely re-pickable, same pattern as authoring's run list, auto-defaulting to the most recent sim); a Settings tab (first, default-selected) shows a viewed sim's full name/description and its fork spec's raw parent run id, hidden while authoring. Two small polish passes landed in the same arc: a `FlowVersionSimsList` display fix (long descriptions were pushing the date off the row — now two independently-clamped lines instead of one shared block), and the reused-step switch showing (disabled) while browsing instead of only appearing during authoring.

## Sims mode (built — design narrative kept for the "why")

**Reuse target correction, settled in discussion**: this does _not_ reuse the old `RunnerSimSelector`/`Sims.tsx`/`CreateSim.tsx`/`ViewSim.tsx`/`SimsFlowView` components (as this doc previously assumed) — those are old-UI pieces bound for their own redesign, and today's step-reuse selection there is genuinely blind (toggles a step by ID/color on the static flow-def graph, with zero visibility into the parent run's actual output/exports). Instead, this reuses the _new_ flow-version-workspace pieces Run History just built: `FlowVersionRunHistoryList`, `useRunEventsWithStatus`, `FlowGraph`+`stepRunInfo`, `StepResultsTab` — the same "reuse is not the goal" distinction applies app-wide, delineating old components not worth carrying forward vs. new workspace pieces actively being built and reused hard.

**Backend needs no schema changes for the minimal version**: `Sim` already has a required, indexed `flowVersionId` FK (`schema.prisma`) — the 1:1 sim-to-flow-version association already exists. `ForkSpec` (`packages/types/src/engine/fork-spec.type.ts`) is already exactly `{parentRunId, reuse: string[]}` — the minimal "reuse steps from one parent run" capability the current system actually supports. The only backend gap is `listSimsByFlowVersionId`, essentially a copy of `listRunsByFlowVersionId`'s shape (port/service/route/adapter/tests). (Separately, whether fork-spec/sim data should eventually move from a CAS artifact to queryable SQL rows is a real, still-open question — deliberately deferred, not blocking this work; see `docs/todo.md`.)

**Left pane: page-state-driven content swap, not tabs.** A bare "Runs" tab would raise a real state-machine question (disabled? hidden? until when?) precisely because Runs isn't a co-equal destination — it only matters while creating a new sim. Model it as page state instead, extending the same pattern `flowVersionRunHistorySlice` already established.

**Corrected during PR 2 planning: 2 modes, not 3 — picking a run is never a one-way gate.** Originally sketched as a linear `browsing → picking-run → authoring` flow where picking a run locked you into authoring for it. Rejected: authoring needs to support freely switching between runs _while_ authoring — comparing what different runs produced — with in-progress reused-step selections and the currently-inspected step (both PR 3) surviving the switch. So:

- **browsing** → left pane shows the sim list (filtered by flowVersionId), with a "New" button
- **authoring** → left pane becomes the run list (`FlowVersionRunHistoryList`, reused as-is) and _stays_ there for the whole time you're authoring — not swapped away once a run's picked. `selectedRunId` is just a freely-updatable field within this mode, not a transition trigger; re-picking a different run updates the main content live. Main content becomes the graph+details view for marking/inspecting reused steps (PR 3); Save or Cancel returns to browsing.

**Interaction: node click means only "inspect," never double duty.** Marking a step reused needs a separate, explicit control — a toggle inside the details panel once a step is inspected, rather than overloading click. Built with shadcn `Switch` (not `Toggle` as originally planned — functionally the same idea, `Switch` just reads clearer for a plain on/off, and is what actually got installed). Right-click (`onNodeContextMenu`) is a clean, cheap option to add later if wanted — React Flow already exposes it as a genuinely distinct prop from `onNodeClick`, requiring only one small additive (non-breaking) prop on `FlowGraph` — but not needed for a first version, especially since this interaction is expected to get revisited once a visual flow editor exists (see `docs/todo.md`'s note on that longer-held, unscheduled goal) and nodes potentially become custom/richer.

**Authoring always shows all 3 panels, even before a run's picked — corrected during PR 3a review.** Originally the "no run selected yet" state showed a 2-panel idle-prompt layout instead of the real graph/details panels. Changed: the graph/details panels already handle empty data gracefully (no `stepRunInfo` just means unstyled nodes, `StepResultsTab` already handles `stepId === null` with its own prompt) — so there's no real reason to special-case that state with a different layout. `FlowVersionSims.tsx` now always renders 3 panels once `mode === "authoring"`, with a small "Select a run below" hint shown above the run list until one's picked.

**Where Cancel (and Save) live — moved out of the details panel, up into the left panel, above the run list.** First attempt put Cancel in the details panel's header; reconsidered because it read as context-less — not obvious _what_ you were cancelling. Left panel now shows a persistent "New Sim" header + Cancel button above the run list for the entire authoring session, regardless of whether a run's picked yet. **Sim Settings (name + description) should land here too, not as a details-panel tab as originally sketched** — same reasoning: a lone Save button in a details-panel tab would have the identical "naked, no context" problem Cancel had. Putting name/description + Save/Cancel together in the left panel, under a "New Sim:" heading, keeps a constant visual anchor for the whole authoring session ("you are creating a sim," not just "here's a random toggle"), and the buttons read as acting on something specific instead of floating alone. Built in PR 3b.

**PR 3b validation/error UI is deliberately minimal — real toast/notification system is future work.** Save requires a non-empty name, a selected run, and at least one reused step (a sim with an empty `reuse` array does nothing — enforced client-side only, `post.ts` doesn't reject it server-side, and it's not yet decided whether that validation belongs at the API layer too). Errors from a failed save render as plain inline text next to the Save/Cancel row — no toast library exists anywhere in `apps/web-app` yet. A shadcn `Tooltip` was installed while exploring a richer treatment for the reused-step hint but held off on — the plan is to revisit inline validation/error rendering once there's a unified toast/notification pattern, rather than build one-off treatments per form. Tracked in `docs/todo.md`.

**Corrected during PR 4 planning: browsing _is_ viewing — not a third mode.** First pass at PR 4 proposed a `"viewing"` mode mirroring how `"authoring"` swaps the left pane to the run list. Rejected: there's no separate viewing mode, because browsing already _is_ viewing. Exactly like authoring keeps the run list visible and lets you freely re-pick `selectedRunId` (a field, not a mode transition), browsing keeps the sim list visible and lets you freely re-pick a new `selectedSimId` field — same pattern, still only 2 modes (`"browsing" | "authoring"`). The middle/right panels stopped branching on `isAuthoring` entirely as a result — they always render now, fed by `inspectingRunId`/`inspectingReusedStepIds` (authoring's own state, or a viewed sim's `parentRunId`/`reuse` fetched via `useGetSimQuery`). Browsing also auto-selects the most recent sim instead of showing nothing, which is what makes Save land you back viewing your just-created sim automatically (`resetToBrowsing` nulls `selectedSimId`, and the list's auto-select effect picks the new latest one up immediately).

PR order:

1. **Backend**: `listSimsByFlowVersionId` — small, precedented, no design risk. Merged (#265).
2. **Scaffold**: the page + left-pane state machine above, sim list ↔ run list swap wired to real data. Merged (#266).
   3a. **Real graph/details wiring**: `FlowGraph`+`useRunEventsWithStatus`+`StepResultsTab` reused for inspection, the reused-step toggle (persisting across run switches, per the state-machine correction above), always-3-panels layout, Cancel anchored in the left panel. Merged (#267).
   3b. **Sim Settings + Save**: name/description inputs anchored in the left panel per the placement decision above, `POST /api/sims`, `sims-api.ts` cache tags so the browsing list picks up new saves. Split out from the original single PR 3 during planning — genuinely separable from 3a, and 3a alone was already a full unit of work. Merged (#268).
3. **View existing sim**: browsing-is-viewing per the correction above, plus a Settings tab (first, default-selected, hidden while authoring) showing the viewed sim's full name/description and raw parent run id. Merged (#269).

## Other bits of work (less fleshed out — sizes and shapes still provisional)

- **Artifacts mode** — real backend piece required, not just a UI tab; more nuanced than the one-line stub this used to be. Core idea, settled: give someone a space to say "I want to make an artifact to use with this flow" (the way run params already work today, and later step-output mocking) — holding artifact metadata plus an explicit association to a flow entity, rather than picking from every artifact that's ever existed globally (today's only option). That curated, associated set is the actual payoff: it's what Sims/Run mode's param-selection UI would draw from instead of the full global list.
  - **Association scope, three tiers, all optional at the schema level**: an artifact can be flagged as belonging to one specific flow version, to a whole flow (all its versions, via `flowId` rather than `flowVersionId`), or globally available. Plus finer-grained param-level curation (which specific artifacts are offered for a specific named param). Schema for all of this is done — `docs/adr/0002-artifact-flow-association-schema.md`, merged (#272): `Artifact` gains nullable `flowId`/`flowVersionId`, settable only through a dedicated association path (never the worker's content-put path — enforced at the type level, not by convention); a new join table curates which artifacts are offered for a specific flow version's named param. Flow params themselves stay unmodeled in SQL (bare-string reference, consistent with how `stepId`/export names already work).
  - **v1 scope, settled: user-created artifacts only.** Not run-produced ones, not promoting a run's export into a curated artifact — both real future ideas, not this pass.
  - **Browsing "user-made artifacts" filters on a dedicated `curated` flag (`docs/adr/0003-artifact-curated-flag.md`), not on `flowId`/`flowVersionId` being set.** First pass at this leaned on "has `flowVersionId` set" as a proxy for user-authorship — workable, since ADR-0002's write-restriction was already shaped for exactly this, but it made those columns carry two jobs at once (curation scope, and now also an authorship signal), which would've quietly broken the moment the system ever had a reason to set `flowVersionId` on a run-produced artifact for its own purposes (e.g. query convenience, since it's already derivable transitively through `Run.flowVersionId`). ADR-0003 decouples this with one small additive column instead: `curated: Boolean @default(false)`, excluded from `ArtifactIndexInput` the same way `flowId`/`flowVersionId` already are, settable only through `associateArtifact`. The create flow still **always** associates a new artifact with the flow version currently being viewed, non-optionally — that part didn't change — but the browse filter is now simply `curated = true`, independent of whatever `flowId`/`flowVersionId` come to mean later. No future wrinkle to track this time: since the worker can never touch `curated` and `put()` already preserves untouched columns across repeated calls on the same hash (verified directly against the real Prisma client while designing this), a run later producing byte-identical content to something already curated cannot un-curate it — confirmed, not assumed.
  - **v1 layout: 3 panels.** Left: browse user-made (`curated = true`) artifacts (list, filterable later, not v1). Middle: selected artifact's content. Right: metadata (label, etc.) and associations (flow version — always set, non-optional at creation per above — and optionally one or more curated params).
  - **Creation UI — leaning tabs-in-page, not fully settled.** Reuse the existing "focused content" tab mechanism (`FlowVersionRunGraphPanel`, already built for Run/Run History/Sims) to open a new-artifact authoring tab in the middle panel, rather than a modal. Reasoning: fits this workspace's own stated IDE-like philosophy better than a modal would, and reuses proven machinery instead of introducing a second UI paradigm just for this. The longer-term "author an artifact from anywhere in the app" idea (e.g. a `+` next to a run param on the Run page, opening the same authoring UI) doesn't require a modal to eventually support — a shared drawer/panel extracted from whatever gets built here would serve the same purpose. Likely needs its own small browsing/authoring state machine, same shape as Sims mode's (`"browsing" | "authoring"`, `docs/UI_WORKSPACE_MILESTONE.md`'s Sims section above) — exact shape to be worked out when this PR is actually planned, not fully speced now.
  - **Editing an existing artifact's metadata/associations — leaning explicit Edit button (view by default → click Edit → fields become editable + Save/Cancel appear), not ambient/always-editable-with-conditional-buttons.** More conventional for "editing an existing record," and association changes have real downstream consequences (what shows up in a future curated picker) — a deliberate "I am now changing this" gesture seems safer than editability that could fire from a stray click.
  - **Explicitly deferred, not v1**: flow-wide/global artifact creation via the UI (schema supports it, create flow won't expose it yet); advanced browsing/filtering (e.g. "all artifacts a step output produces," "all run params"); side-by-side artifact comparison/diffing (a real future want, genuinely unsure if this page is even the right home for it, or something else — not blocking, a separate question).
  - **PR order** (of the Artifacts mode initiative — mirrors how Sims mode's now-completed v1 arc was tracked as PRs 1/2/3a/3b/4, cited purely as precedent for the convention, not implying this is at a similar stage):
    1. **Backend schema**: the `Artifact`/`ArtifactParamCuration` changes above, port + Prisma adapter, no HTTP routes yet. Merged (#272).
    2. HTTP routes for `associateArtifact`/`curateArtifactForParam`/`uncurateArtifactForParam`/`listCuratedArtifacts` — not started.
       3+. The actual page: browse/view (read-only 3-panel layout), then create/author, then edit-existing-metadata — likely separate PRs along those lines, matching the Sims-mode pattern of splitting once each piece's shape is concrete; not planned in detail yet.
  - Estimated smaller than the Evals rework, but not sized precisely yet.
- **Edit mode, actually editable** — currently just a read-only copy of View. Real scope unknown; likely a substantial lift given flow authoring/validation/publish-state semantics aren't designed yet (e.g. whether a published version becomes immutable and editing forks a new draft version).
- **A full-step "resolved preview" view — parked, not scoped.** Raised while building Step Results: a view showing _every_ field of a step (matching `StepHttpJsonDetails`' full field list — URL/Method/Headers/Body/Exports — not just fields with refs), with each field showing its bound param value before a run and its actual resolved value after. Two distinct gaps motivate this, not one:
  - Fields with zero refs are invisible in Run mode entirely — only Edit/View's `StepHttpJsonDetails` shows them, and only unresolved.
  - Fields that _do_ have refs are only ever shown at leaf-`bindPath` granularity in Field Resolution (e.g. just `body.messages[1].content`), never composed back up into the whole logical field (`body`, `headers`) as it was actually sent — if `body` has refs at two different nested paths, Field Resolution shows two disjoint leaf rows, never one composed `body` object with both substitutions sitting in their natural positions. The natural grain for "the whole field, fully resolved" is the same field-level granularity `StepHttpJsonDetails` already uses; Field Resolution deliberately went finer because "what did this one reference become" wants the leaf, not the whole object.

  Also explicitly not Run-History-specific, and connects to an existing feature, not just Field Resolution: the per-param "show usages" report (`findParamRefs`/`renderParamRefReport`, `FlowVersionRunParamRow`) already lets you click a _param_ and see a markdown report of everywhere it's used — useful, but it's the reverse direction (param → its usages, leaf-level, textual report). What's missing is the other direction: click a _step_ in the live Params panel and see that whole step's fields composed with the currently-selected param artifacts filled in, ahead of submitting — same underlying resolution logic (`resolveRefHash`/`buildRefUsage`/`foldResolvedField`), organized by step instead of by param, at whole-field instead of leaf granularity. Deliberately not built yet because it's unclear how much it'd repeat Field Resolution/References; the resolution reached in discussion: judge that by whether it answers a _different question_ (a holistic "what did this step look like as one document" snapshot vs. drill-down provenance), not by whether it reuses the same underlying resolved-value logic — reusing `foldResolvedField` etc. across views isn't repetition, showing the same field list with the same framing twice would be. Needs its own design pass before it's scope, the same way Step Results got one.

- **Replay — acknowledged valuable, not scheduled (not "no reason exists" anymore).** Watching a historical run's execution animate at some controlled pace (a play-head over the event timeline instead of a range selection, reconstructing state at any point rather than just the final one) — technically straightforward given the architecture (`deriveStepRunInfo` already works on any event array, live or historical; this is UI/timing work, not new plumbing). The concrete reason it's worth building, not just "it'd be cool": it lets you re-experience the feel of watching a run live — pacing, what happened when — which a final-state view structurally can't reproduce. Still not the first thing to build; no committed timing.

## Evals: deliberately its own separate, future milestone

Explicitly not part of this arc's near-term scope — this is a real data-model rearchitecture, not a UI tab, and deserves its own milestone the same way the original eval vertical slice did (Milestone #9). Current state: evals are embedded inside the flow definition itself (an eval is just a flow with a judge step) — a smaller move than originally wanted. The actual goal: evals as standalone, reusable entities, decoupled from any one flow — a fixed template (System Prompt + Question + Context + Answer) that can be constructed and run against different flows. Inside this workspace, an Evals mode/tab would let you construct an eval run (bind the flow's actual output to the eval template's expected inputs) — results visualization might end up needing its own separate page if it doesn't fit inside that tab. Not scoped further here; needs its own dedicated design pass.

## Process note

Milestones here don't need to map 1:1 to version bumps or a formal GitHub Milestone object — each chunk above can close independently, with version bumps happening at whatever later point represents a real, demo-able jump (e.g. "the revamped UI once the pages are mostly done"), not tied to milestone count.
