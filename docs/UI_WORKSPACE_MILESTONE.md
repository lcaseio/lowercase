# Flow-Version Workspace UI Rework

## Summary

This is the direction doc for the ongoing rework of `apps/web-app` away from a page-navigation web app toward a flow-version-centric workspace: drill into one flow version, and work with it entirely through panels/tabs for whichever mode you're in (Edit, View, Run, Run History, Sims, Artifacts, Evals) — rather than navigating to separate top-level pages for runs/sims/artifacts/evals that all have to be manually cross-referenced by id.

Not everything below is equally fleshed out. The point of writing it now is to capture the parts that _are_ settled (so they don't need re-deriving later) and name the parts that aren't (so they read as open, not forgotten). This doesn't map to a single GitHub Milestone or version bump — each concrete chunk below (starting with Run History) is its own small, closeable milestone; this doc is the throughline connecting them.

## Overall idea and purpose

The core problem with the current page-based UI: nearly everything lives at the "list all runs / list all sims / list all artifacts" level, globally, with no real organizing entity underneath it. That's the wrong default — the actual daily unit of work is _one specific flow version_. The rework makes that the primary workspace:

- **Flow version = the primary workspace.** Drill into one, and within that single context: edit it (if still a draft), run it, view its run history, create/view sims of it, browse/create artifacts scoped to it, construct and run evals against it — all via panels/tabs inside one persistent shell, VS-code-style (frames with tabs, panels loading different content).
- **Flow (all versions) = one level up.** Comparing sims/runs/artifacts/evals across every version of one flow — needed, but not the primary workspace.
- **Global (all flows/all runs/etc.) = top, rare.** Useful sometimes as a filterable escape hatch, but explicitly not where work usually happens — today's UI is almost entirely this level, which is the concrete reason it's been "not very useful."

Longer-term context worth keeping in view even though it's not being built yet: the eventual target runtime is Tauri (not just a browser tab), with real multi-pane coexistence (view + edit + preview at once, like an IDE) rather than one panel at a time, and an explicit non-goal of supporting small/phone-sized screens — the panel/toolbar model is inherently a desktop pattern. Also an explicit constraint: this UI is being hand-designed and understood, not generated wholesale by an AI/design tool in one shot, the same ownership stance as the rest of this codebase.

## What's already built

- The routing shell: `/spike/:flowId/:versionId/<mode>` via `FlowVersionWorkspace` (real flow-version fetch) + `FlowVersionModeNav`, replacing the old flat `Spike.tsx` prototype. One persistent shell, one `Outlet` per mode, state scoped per mode rather than globally.
- **Edit / View modes**: routed and rendering (flow graph + JSON view, step details, settings/params/problems panels), but not yet functionally distinct from each other — Edit isn't actually editable yet. Matches the concrete panel pairing already envisioned: flow graph + JSON definition for authoring context.
- **Run mode**: fully working end to end — submit a run, watch it live (websocket-fed events, no polling), an animated flow chart showing live step status (pending/running/completed/failed) derived client-side from the run's own event stream, a params panel with artifact selection, and a preview feature showing how a selected param actually resolves everywhere it's referenced in the flow (including refs embedded in larger template strings). Matches the other envisioned panel pairing: flow graph + step execution detail for run/inspection context.
- **Step Results** (Run mode's right panel, PR #261): status, Output & Exports, Field Resolution (a field's final value with every `{{...}}` ref in it substituted, composed across multiple refs in one field), and References (each individual reference's own resolved value) — all sourced from the run's own event stream, deliberately with zero dependency on `job.<cap>.submitted`'s payload shape (see `docs/todo.md`).
- Reusable pieces this unlocked for later modes: `EventGraph` (decoupled from any one page's state, takes events/selection as plain props), `deriveStepRunInfo`/`useStepRunInfo` (a pure fold over any event array, not tied to live-only data — status plus output/export hashes), `findStepRefs`/`resolveRefHash`/`foldResolvedField` (`lib/ref-resolution.ts`, also event-array-agnostic).
- **Run History mode**: fully built across 3 small PRs (#262, #263, #264) — a flow-version-filtered run list (`listRunsByFlowVersionId`), a left-panel run list wired to routing with REST event backfill on selection, and Run mode's own graph/details panels (`FlowVersionRunGraphPanel`/`FlowVersionRunDetailsPanel`, `StepResultsTab`) reused as-is, fed by the selected historical run's events instead of the live one — confirmed those panels needed zero changes, since they were already pure props-in/callbacks-out with no coupling to the live Redux slice. Also produced two small shared extractions used by both Run and Run History: `useRunEventsWithStatus` (events backfill + `deriveStepRunInfo`, identical logic regardless of live vs. historical) and `lib/run-panel-state.types.ts` (tab/focused-content types hoisted out of the live slice, since they were never actually live-specific — each mode still keeps its own independent slice state, no shared/merged state between modes).
- **Sims mode**: fully built across 5 PRs (#265, #266, #267, #268, #269) — see "Sims mode" section below for the full design narrative (state machine, placement decisions, corrections). Summary: authoring a new sim reuses Run History's graph/details pieces with an added reused-step toggle; browsing _is_ viewing (the sim list stays visible and freely re-pickable, same pattern as authoring's run list, auto-defaulting to the most recent sim); a Settings tab (first, default-selected) shows a viewed sim's full name/description and its fork spec's raw parent run id, hidden while authoring. Two small polish passes landed in the same arc: a `FlowVersionSimsList` display fix (long descriptions were pushing the date off the row — now two independently-clamped lines instead of one shared block), and the reused-step switch showing (disabled) while browsing instead of only appearing during authoring.

## Sims mode (built — design narrative kept for the "why")

**Reuse target correction, settled in discussion**: this does _not_ reuse the old `RunnerSimSelector`/`Sims.tsx`/`CreateSim.tsx`/`ViewSim.tsx`/`SimsFlowView` components (as this doc previously assumed) — those are old-UI pieces bound for their own redesign, and today's step-reuse selection there is genuinely blind (toggles a step by ID/color on the static flow-def graph, with zero visibility into the parent run's actual output/exports). Instead, this reuses the _new_ flow-version-workspace pieces Run History just built: `FlowVersionRunHistoryList`, `useRunEventsWithStatus`, `FlowGraph`+`stepRunInfo`, `StepResultsTab` — the same "reuse is not the goal" distinction applies app-wide, delineating old components not worth carrying forward vs. new workspace pieces actively being built and reused hard.

**Backend needs no schema changes for the minimal version**: `Sim` already has a required, indexed `flowVersionId` FK (`schema.prisma`) — the 1:1 sim-to-flow-version association already exists. `ForkSpec` (`packages/types/src/engine/fork-spec.type.ts`) is already exactly `{parentRunId, reuse: string[]}` — the minimal "reuse steps from one parent run" capability the current system actually supports. The only backend gap is `listSimsByFlowVersionId`, essentially a copy of `listRunsByFlowVersionId`'s shape (port/service/route/adapter/tests). (Separately, whether fork-spec/sim data should eventually move from a CAS artifact to queryable SQL rows is a real, still-open question — deliberately deferred, not blocking this work; see `docs/todo.md`.)

**Left pane: page-state-driven content swap, not tabs.** A bare "Runs" tab would raise a real state-machine question (disabled? hidden? until when?) precisely because Runs isn't a co-equal destination — it only matters while creating a new sim. Model it as page state instead, extending the same pattern `flowVersionRunHistorySlice` already established.

**Corrected during PR 2 planning: 2 modes, not 3 — picking a run is never a one-way gate.** Originally sketched as a linear `browsing → picking-run → authoring` flow where picking a run locked you into authoring for it. Rejected: authoring needs to support freely switching between runs _while_ authoring — comparing what different runs produced — with in-progress reused-step selections and the currently-inspected step (both PR 3) surviving the switch. So:

- **browsing** → left pane shows the sim list (filtered by flowVersionId), with a "New" button
- **authoring** → left pane becomes the run list (`FlowVersionRunHistoryList`, reused as-is) and _stays_ there for the whole time you're authoring — not swapped away once a run's picked. `selectedRunId` is just a freely-updatable field within this mode, not a transition trigger; re-picking a different run updates the main content live. Main content becomes the graph+details view for marking/inspecting reused steps (PR 3); Save or Cancel returns to browsing.

**Interaction: node click means only "inspect," never double duty.** Marking a step reused needs a separate, explicit control — a toggle inside the details panel once a step is inspected, rather than overloading click. Built with shadcn `Switch` (not `Toggle` as originally planned — functionally the same idea, `Switch` just reads clearer for a plain on/off, and is what actually got installed). Right-click (`onNodeContextMenu`) is a clean, cheap option to add later if wanted — React Flow already exposes it as a genuinely distinct prop from `onNodeClick`, requiring only one small additive (non-breaking) prop on `FlowGraph` — but not needed for a first version, especially since this interaction is expected to get revisited once a visual flow editor exists (see `docs/todo.md`'s note on that longer-held, unscheduled goal) and nodes potentially become custom/richer.

**Authoring always shows all 3 panels, even before a run's picked — corrected during PR 3a review.** Originally the "no run selected yet" state showed a 2-panel idle-prompt layout instead of the real graph/details panels. Changed: the graph/details panels already handle empty data gracefully (no `stepRunInfo` just means unstyled nodes, `StepResultsTab` already handles `stepId === null` with its own prompt) — so there's no real reason to special-case that state with a different layout. `FlowVersionSims.tsx` now always renders 3 panels once `mode === "authoring"`, with a small "Select a run below" hint shown above the run list until one's picked.

**Where Cancel (and Save) live — moved out of the details panel, up into the left panel, above the run list.** First attempt put Cancel in the details panel's header; reconsidered because it read as context-less — not obvious _what_ you were cancelling. Left panel now shows a persistent "New Sim" header + Cancel button above the run list for the entire authoring session, regardless of whether a run's picked yet. **Sim Settings (name + description) should land here too, not as a details-panel tab as originally sketched** — same reasoning: a lone Save button in a details-panel tab would have the identical "naked, no context" problem Cancel had. Putting name/description + Save/Cancel together in the left panel, under a "New Sim:" heading, keeps a constant visual anchor for the whole authoring session ("you are creating a sim," not just "here's a random toggle"), and the buttons read as acting on something specific instead of floating alone. Built in PR 3b.

**PR 3b validation/error UI is deliberately minimal — real toast/notification system is future work.** Save requires a non-empty name, a selected run, and at least one reused step (a sim with an empty `reuse` array does nothing — enforced client-side only, `post.ts` doesn't reject it server-side, and it's not yet decided whether that validation belongs at the API layer too). Errors from a failed save render as plain inline text next to the Save/Cancel row — no toast library exists anywhere in `apps/web-app` yet. A shadcn `Tooltip` was installed while exploring a richer treatment for the reused-step hint but held off on — the plan is to revisit inline validation/error rendering once there's a unified toast/notification pattern, rather than build one-off treatments per form. Tracked in `docs/todo.md`.

**Corrected during PR 4 planning: browsing _is_ viewing — not a third mode.** First pass at PR 4 proposed a `"viewing"` mode mirroring how `"authoring"` swaps the left pane to the run list. Rejected: there's no separate viewing mode, because browsing already _is_ viewing. Exactly like authoring keeps the run list visible and lets you freely re-pick `selectedRunId` (a field, not a mode transition), browsing keeps the sim list visible and lets you freely re-pick a new `selectedSimId` field — same pattern, still only 2 modes (`"browsing" | "authoring"`). The middle/right panels stopped branching on `isAuthoring` entirely as a result — they always render now, fed by `inspectingRunId`/`inspectingReusedStepIds` (authoring's own state, or a viewed sim's `parentRunId`/`reuse` fetched via `useGetSimQuery`). Browsing also auto-selects the most recent sim instead of showing nothing, which is what makes Save land you back viewing your just-created sim automatically (`resetToBrowsing` nulls `selectedSimId`, and the list's auto-select effect picks the new latest one up immediately).

PR order:

1. **Backend**: `listSimsByFlowVersionId` — small, precedented, no design risk. Merged (#265).
2. **Scaffold**: the page + left-pane state machine above, sim list ↔ run list swap wired to real data. Merged (#266).
   3a. **Real graph/details wiring**: `FlowGraph`+`useRunEventsWithStatus`+`StepResultsTab` reused for inspection, the reused-step toggle (persisting across run switches, per the state-machine correction above), always-3-panels layout, Cancel anchored in the left panel. Merged (#267).
   3b. **Sim Settings + Save**: name/description inputs anchored in the left panel per the placement decision above, `POST /api/sims`, `sims-api.ts` cache tags so the browsing list picks up new saves. Split out from the original single PR 3 during planning — genuinely separable from 3a, and 3a alone was already a full unit of work. Merged (#268).
3. **View existing sim**: browsing-is-viewing per the correction above, plus a Settings tab (first, default-selected, hidden while authoring) showing the viewed sim's full name/description and raw parent run id. Merged (#269).

## Other bits of work (less fleshed out — sizes and shapes still provisional)

- **Artifacts mode** — real backend piece required, not just a UI tab; more nuanced than the one-line stub this used to be. Core idea, settled: give someone a space to say "I want to make an artifact to use with this flow" (the way run params already work today, and later step-output mocking) — holding artifact metadata plus an explicit association to a flow entity, rather than picking from every artifact that's ever existed globally (today's only option). That curated, associated set is the actual payoff: it's what Sims/Run mode's param-selection UI would draw from instead of the full global list.
  - **Association scope, three tiers, all optional at the schema level**: an artifact can be flagged as belonging to one specific flow version, to a whole flow (all its versions, via `flowId` rather than `flowVersionId`), or globally available. Plus finer-grained param-level curation (which specific artifacts are offered for a specific named param). Schema for all of this is done — `docs/adr/0002-artifact-flow-association-schema.md`, merged (#272): `Artifact` gains nullable `flowId`/`flowVersionId`, settable only through a dedicated association path (never the worker's content-put path — enforced at the type level, not by convention); a new join table curates which artifacts are offered for a specific flow version's named param. Flow params themselves stay unmodeled in SQL (bare-string reference, consistent with how `stepId`/export names already work).
  - **v1 scope, settled: user-created artifacts only.** Not run-produced ones, not promoting a run's export into a curated artifact — both real future ideas, not this pass.
  - **Browsing "user-made artifacts" filters on a dedicated `curated` flag (`docs/adr/0003-artifact-curated-flag.md`), not on `flowId`/`flowVersionId` being set.** First pass at this leaned on "has `flowVersionId` set" as a proxy for user-authorship — workable, since ADR-0002's write-restriction was already shaped for exactly this, but it made those columns carry two jobs at once (curation scope, and now also an authorship signal), which would've quietly broken the moment the system ever had a reason to set `flowVersionId` on a run-produced artifact for its own purposes (e.g. query convenience, since it's already derivable transitively through `Run.flowVersionId`). ADR-0003 decouples this with one small additive column instead: `curated: Boolean @default(false)`, excluded from `ArtifactIndexInput` the same way `flowId`/`flowVersionId` already are, settable only through `associateArtifact`. The create flow still **always** associates a new artifact with the flow version currently being viewed, non-optionally — that part didn't change — but the browse filter is now simply `curated = true`, independent of whatever `flowId`/`flowVersionId` come to mean later. No future wrinkle to track this time: since the worker can never touch `curated` and `put()` already preserves untouched columns across repeated calls on the same hash (verified directly against the real Prisma client while designing this), a run later producing byte-identical content to something already curated cannot un-curate it — confirmed, not assumed.
  - **v1 layout: 3 panels.** Left: browse user-made (`curated = true`) artifacts (list, filterable later, not v1). Middle: selected artifact's content. Right: metadata (label, etc.) and associations (flow version — always set, non-optional at creation per above — and optionally one or more curated params).
  - **Creation UI — leaning tabs-in-page, not fully settled, but considerably more concrete after discussion.** A new artifact opens as its own content tab in the middle panel (reusing the existing "focused content" tab mechanism from `FlowVersionRunGraphPanel`, already built for Run/Run History/Sims), rather than a modal — fits this workspace's IDE-like philosophy and reuses proven machinery instead of a second UI paradigm just for this. Two distinct entry points, not one form, since upload and in-editor authoring are genuinely different inputs (a file vs. a format+typed-content pair) and binary content can only ever be uploaded: **"Add File"** (multipart upload — deliberately not calling it "upload" in the UI, thinking ahead to a desktop feel; already backed by `post-artifact-file.ts`) vs. **"Write Artifact"** (in-editor authoring — only JSON has a dedicated route today, `put-json-artifact.ts`; text/markdown authoring-in-editor has no route yet, upload-only, a small backend gap to close when this is actually built). Content format for authoring isn't asked upfront — borrowing VS Code's own answer for "what kind of new file is this": open with a sensible default, let format be a small selector changeable at any time while authoring, only really mattering at save. Unsaved/new-artifact state (both "is this new" and "what happens if I navigate away with unsaved changes") reuses the same tab affordance rather than inventing new UI — a dirty-indicator dot on the tab and a confirm-before-navigating-away prompt, the same pattern VS Code already uses for unsaved file tabs. Still needs its own small browsing/authoring state machine, same shape as Sims mode's (`"browsing" | "authoring"`) — exact implementation shape to be worked out when this piece is actually planned (see PR order below), not fully speced now. The longer-term "author an artifact from anywhere in the app" idea (e.g. a `+` next to a run param on the Run page, opening the same authoring UI) doesn't require a modal to eventually support — a shared drawer/panel extracted from whatever gets built here would serve that too.
  - **Editing an existing artifact's metadata/associations — leaning explicit Edit button (view by default → click Edit → fields become editable + Save/Cancel appear), not ambient/always-editable-with-conditional-buttons.** Sharper reasoning after discussion: this isn't a "desktop vs. web" question (direct-manipulation tools like Premiere/Resolve don't gate edits behind a mode either) — it's a usage-frequency question. Those tools assume you're almost always actively manipulating the one thing you're focused on; this panel is closer to a file browser with an inspector attached, where most visits are "what is this" and actual edits are occasional and deliberate, at least until real usage says otherwise. Association changes also have real downstream consequences (what shows up in a future curated picker), so a deliberate "I am now changing this" gesture fits. Cheap to reverse later — flip the default if editing turns out far more frequent than browsing once there's real content to work with.
  - **Explicitly deferred, not v1**: flow-wide/global artifact creation via the UI (schema supports it, create flow won't expose it yet); advanced browsing/filtering (e.g. "all artifacts a step output produces," "all run params"); side-by-side artifact comparison/diffing (a real future want, genuinely unsure if this page is even the right home for it, or something else — not blocking, a separate question).
  - **PR order** (of the Artifacts mode initiative — mirrors how Sims mode's now-completed v1 arc was tracked as PRs 1/2/3a/3b/4, cited purely as precedent for the convention, not implying this is at a similar stage). Using bold labels rather than a markdown ordered list deliberately — a numbered list here previously got silently mis-renumbered by the formatter once 4a/4b were inserted:
    - **PR 1 — Backend schema**: the `Artifact`/`ArtifactParamCuration` changes above, port + Prisma adapter, no HTTP routes yet. Merged (#272), plus the `curated` flag (#273).
    - **PR 2 — HTTP routes**: `associateArtifact`/`curateArtifactForParam`/`uncurateArtifactForParam`/`listCuratedArtifacts`, a filterable `GET /api/artifacts` (`flowId`/`flowVersionId`/`curated` query params), and a generic `PATCH /api/artifacts/:hash`. Merged (#274).
    - **PR 3 — Page scaffold + browse**: the 2-panel shell (list | content), a real list of this flow version's `curated = true` artifacts on the left, click-to-view an artifact's content in the middle panel. No metadata/edit panel yet — a list with nothing to click into isn't a real testable slice on its own, so list + content view are one PR, not two. Merged (#275).
    - **PR 4a — `GET /api/artifacts` rework + list decoration**: split out of what was originally planned as "PR 4" (the metadata-inspect panel) — showing per-param curation turned out to be more useful on the _list_ than in a per-artifact detail panel, and cheaper (a query scoped only to `flowVersionId`, already indexed, versus a per-artifact reverse lookup that didn't exist). `GET /api/artifacts` now returns `ArtifactListItem[]` (`{artifact, associations: {flowId, flowVersionId, curated, paramCurations}}`) instead of flat `ArtifactIndex[]` — a deliberate breaking change to an already-shipped endpoint, with every consumer (including the old flat `/artifacts` page and the Run/Runner param pickers) updated to match. Also added a formal Prisma relation from `ArtifactParamCuration.artifactHash` to `Artifact.hash` (previously bare-string, an oversight from PR 2 rather than a considered decision — see the corrected ADR-0002 read) — a real FK constraint, not just conceptual tidiness. Merged (#276).
    - **PR 4b — Metadata/associations, read-only**: right panel renders full metadata for whatever's selected in the middle. Not started; two design questions already resolved during an earlier (then-reoriented) planning pass, worth keeping rather than re-deriving:
      - **Build a small standalone panel, don't reuse `FlowVersionRunDetailsPanel`'s tabs.** That component is tightly coupled to run/sim/step concepts (`selectedEvent`, `stepRunInfo`, `paramHashes`, `flowDef`); `FlowVersionArtifacts.tsx` currently has zero tab mechanism, and bolting artifact concerns onto the run/sim panel would widen its coupling for no real benefit.
      - **Visual style is a mix of both precedents, split by field kind, not a single wholesale choice of one over the other. Field order follows that same split — uneditable identity fields first, then the editable group (`label`, then the "shared across versions" toggle, then the param checkbox list), not interleaved:**
        - Identity fields that will never become editable (`hash`, `time`, `contentType`, `size`, `format`) use `SimSettingsTab`'s plain label/value block style — they're a summary, not a form.
        - Fields that _are_ going to be editable in PR 5 (`label`, and any others that turn out editable) render now as read-only form fields — `StepHttpJsonDetails`/`InputField`'s read-only-`<Input>`-row style — the same "shape now, wire later" reasoning `InputField` already uses, just extended to a field that happens to start read-only.
        - **Param associations render as a checkbox list, not a label/value pair or plain param-name text.** Source the checkbox set from the flow version's declared params (`definition.params`, the same source `curateArtifactForParam` already validates against), checked state from `associations.paramCurations`. No existing Fields component does a checkbox list today (`fields/` has `InputField`/`TextAreaField`/`HeadersField`/`ExportsField`/`InputListField`/`CodeEditorField`/`EvalContext*Field`, none checkbox-based) — this is a new one, `disabled` for now like the rest.
        - **The checkbox list is further filtered to only type-compatible params, not every declared param.** Reuses `isArtifactCompatible` (`packages/flow-analysis/src/artifact-compat.ts`) — the same check `FlowVersionRunParamRow`/`RunnerParamRow` already use client-side and `run.service.ts` uses server-side to validate a run's param artifacts — so an artifact never shows as offerable for a param whose declared type it doesn't match. Surfaced a real backend gap while planning this: `curateArtifactForParam` itself never runs this check, unlike run submission — logged in `docs/todo.md`, fix belongs in PR 5 (the write path), not this one.
        - **New field, not previously in this doc: a "usable across flow versions" toggle, labeled just "Share" in the UI.** This isn't a new backend concept — it's exposing the `crossVersion` flag `ArtifactService.curateArtifactForParam` (`packages/services/src/artifact.service.ts:56`) already accepts, which sets the artifact's `flowId` (not just `flowVersionId`) when true. Render with `Switch` (`components/ui/switch`), following `StepResultsTab`'s "Reuse" toggle precedent (`Label` + `Switch` pair, `disabled={!onToggleReused}` when no write callback is wired) — same disabled-without-a-callback pattern this panel needs anyway since it's read-only for now. **Label settled as "Share"**, not the longer "Shared across versions" first considered — a single present-tense verb, mirroring `StepResultsTab`'s "Reuse" (also present-tense-verb, not a state adjective like "Reused") rather than a new convention, and chosen specifically to fit the same `w-20` fixed label column every other field in this panel uses (a longer label wrapped ugly at that width). Less immediately clear on its own; a tooltip/icon is the planned way to recover that clarity later, not a wider label column.

      - **Cross-cutting: Fields components need edit-mode as a prop/callback, not a fork.** `InputField` today is unconditionally `readOnly` (`components/fields/InputField.tsx:27`) — there's no prop that would let the same component serve both PR 4b (read-only) and PR 5 (editable) without duplicating it. Whatever gets built for 4b — the new checkbox-list field and the `Switch` toggle included — should take editability the same way `StepResultsTab`'s `Switch` already does: presence of an `onChange`/`onCommit`-style callback controls whether the control is live, absence makes it read-only/disabled. Worth fixing this in the shared `InputField` et al. (or at least designing new fields this way from the start) now, so PR 5 wires callbacks into existing components instead of swapping them out.

      - **Forward-looking, not deciding now: PR 6+'s authoring flow will likely want to reuse these same field components, not just 4b/5's read/edit ones.** `contentType`/`format` is the specific field this bites — today it's planned as a plain identity value (never editable, per the split above), but authoring needs it selectable (a dropdown), matching the "small selector changeable at any time while authoring" behavior already anticipated for the format picker (see the Creation UI bullet above). Whether that means the same field slot grows a third, select-driven mode, or authoring gets its own separate field, depends on how the authoring flow's format-selector affordance actually gets built — don't lock in an approach in 4b that would foreclose either option.

      - **Resolved: uncurating an artifact from every param on a flow version is fine and doesn't touch `flowId`.** Param curation is optional and independent of version association — an artifact can be associated to a `flowVersionId` with zero curated params. There's also no UI to _un_-associate an artifact from a `flowVersionId` at all, and none is being built now — not a gap, a deliberate non-goal for this pass.
      - **Genuinely open, deferred, not just "not yet designed": how `flowId`-level sharing across versions actually behaves once more UI layers exist.** This is bigger than a display detail (like the per-entry `flowVersionId` question below) — how "shared across versions" should resolve once there's real multi-version curation, editing, and picker UI to react to it may not be answerable from here, and may end up reshaping the schema rather than just the panel. Don't try to fully answer it now; revisit once the layers that would surface the real tension (multi-version curation UI, the param picker actually consuming `flowId`-shared artifacts) exist.
      - **Still open, smaller/display-only: each `paramCurations` entry's own `flowVersionId`, once `crossVersion` curation is visible.** Today's list always filters by one `flowVersionId`, so that field is currently redundant per-entry — but if an artifact ends up curated across versions, the panel may be where that needs to surface (e.g. per-row version labels in the checkbox list above). Not yet designed. Don't implement this yet.

    - **PR 4c — Unify the artifact metadata write path, before PR 5's UI consumes it.** Originally scoped narrower (just fix the missing `label` write path), widened during design into a full consolidation of `associateArtifact`/`curateArtifactForParam`/`uncurateArtifactForParam` into one write path. **Merged (#278).** This is the layer PR 5's Save button actually calls.

      - **Motivating problem: Save composing several independent API calls felt wrong, and tracing why led somewhere better than a frontend workaround.** The friction wasn't cosmetic — `ArtifactService.curateArtifactForParam` today already does multi-step write orchestration itself (calls `associateArtifact`, then a second repository call), meaning the _service_ layer is doing a job that belongs at the _adapter_ layer. Consolidating fixes the actual layering issue, not just the symptom.

      - **Repository-level collapse: one port method instead of three.** `associateArtifact`/`curateArtifactForParam`/`uncurateArtifactForParam` became one `ArtifactRepositoryPort` method, `updateMetadata` (service-layer counterpart: `ArtifactServicePort.updateArtifactMetadata`), taking one `ArtifactMetadata` object. The Prisma adapter decides internally whether a plain `update` call suffices or a `$transaction` is needed — matching the existing precedent in this codebase for "upsert a parent row, replace a full set of related child rows": `createRun`'s handling of `RunParam` (`packages/adapters/src/run-repository/prisma-run-repository.ts:91-148` — `$transaction(async (tx) => {...})`, upsert the `Run`, then `deleteMany`+`createMany` the `RunParam` rows), not a new convention invented for this.

      - **The settled `ArtifactMetadata` type** (destined for `packages/types/src/artifacts/`) — a discriminated union so an invalid combination can't even be constructed, not just runtime-checked:

        ```ts
        type ArtifactMetadata =
          | {
              label?: string | null;
              flowId?: string | null;
              flowVersionId?: string | null;
              paramCurations?: undefined;
            }
          | {
              label?: string | null;
              flowId?: string | null;
              flowVersionId: string;
              paramCurations?: string[];
            };
        ```
        - `label`/`flowId`/`flowVersionId` are all tri-state (`undefined` = leave unchanged, `null` = clear, value = set) — the same convention `ArtifactAssociation` already established for `flowId`/`flowVersionId`, just extended to `label` and made consistent.
        - `label` closes a real, verified gap: `PatchArtifactReq`/`ArtifactAssociation` today is only `{flowId?, flowVersionId?, curated?}` — no `label` field at all, confirmed directly against `packages/types/src/artifacts/artifact-association.ts`. An oversight, not a considered exclusion.
        - `flowVersionId` is one shared top-level field, not duplicated per-`paramCurations`-entry — today's actual behavior (`curateArtifactForParam`) already keeps the artifact's own `flowVersionId` column in sync with whichever version you're curating for, unconditionally, every time. No existing case has them diverge, so the type doesn't need to represent one.
        - `paramCurations` is a flat `string[]` (just param names) — full-replace semantics, scoped to whichever `flowVersionId` is in the same request. Sending the array replaces that version's entire curated-param set (delete what's missing, create what's new); the repository does this unconditionally, no read-first diff needed at the application level, mirroring the `RunParam` full-replace precedent above.
        - The union's second branch requires `flowVersionId` whenever `paramCurations` is present — "curate params but don't say which version" is a type error, not a silently-defaulted or runtime-checked edge case. Preferred deliberately over a lazy fallback-to-the-artifact's-current-`flowVersionId` behavior — that would've been one more implicit behavior to remember and test; treating the omission as a malformed request is simpler and more honest about what the caller actually needs to know.
        - **`curated` is deliberately excluded entirely** — never settable through this type. Once curated, the existing invariant (a run producing byte-identical content can never un-curate it, verified against `put()`'s preserve-untouched-columns behavior) means exposing it as freely settable here would risk breaking that by accident, and no un-curate affordance exists anywhere in this UI's design anyway.
        - Meant to serve both this PATCH-style edit path and, later, PR 6+'s creation flow (bundled alongside initial content) — one shape, not two, reused rather than duplicated.

      - **Settled: the HTTP endpoint is `PATCH /api/artifacts/:hash`, unchanged route, widened body.** No new endpoint needed — this route already exists (`associateArtifact`'s current PATCH); the only change is its body type, from `ArtifactAssociation` to the new `ArtifactMetadata`. REST-correct on its own terms: PATCH's semantics (partial update, explicit clear/leave) already match the tri-state design, not a coincidence.

      - **Settled: the existing param-centric curate/uncurate routes** (`POST`/`DELETE /flows/versions/:versionId/params/:paramName/curated-artifacts`) **get removed, not kept as a second entry path.** Sharper reason than "nothing calls them yet" (confirmed via grep, still true): `:paramName` in that URL addresses something with no real identity in the data model — params are deliberately unmodeled in SQL (`CLAUDE.md`: bare-string keys inside a flow definition, no `Param` row/id), so that path was modeling the wrong thing as the resource. Curation is a property _of the artifact_ ("which params does this serve"), not of a param that happens to have one — `/api/artifacts/:hash` addresses something real (the `Artifact` row); the param-centric path never did.

      - **Checked the unified `/api/artifacts` shape against every other near-future use case before settling, not just this write path** — Run mode's param-picker fix (still unsolved separately) fits as one more `GET /api/artifacts` query filter (`?flowVersionId=X&paramName=Y`), no new resource; PR 6+'s creation routes (`POST .../json`, `POST .../files`) would widen their bodies to accept `ArtifactMetadata` the same way, since it's meant to be shared across create and edit; eventual deletion would be `DELETE /api/artifacts/:hash`, unremarkable. Nothing needed a new resource shape or special-cased path — a decent signal the shape is right, since the previous param-centric design failed exactly this kind of check.

      - **Explicitly not building now: a generic filter DSL for `GET /api/artifacts`.** Raised while sanity-checking the shape — the user can picture a future powerful browsing UI where "curated for this param" is just one of many possible filters, combined flexibly. Today's flat named-optional-fields filter shape (`flowId?`, `flowVersionId?`, `curated?`, soon `paramName?`) scales fine for _more named filters over time_ (purely additive, no breaking change) but wouldn't scale to _arbitrary combinable filter logic_ — that would need a real redesign (likely a structured filter body, not query-string fields) when it's actually being built. Deliberately not designed speculatively now, since a real UI need should dictate that shape, not a guess. Doesn't block anything here: reads and writes are already separate endpoints, so this stays entirely a future read-side concern.

      - **Why this matters beyond tidiness: Run mode's param picker is still fully unfiltered.** Verified: `FlowVersionRunParamsPanel.tsx` calls `useListArtifactsQuery()` with no filter at all, narrowing only by `isArtifactCompatible` client-side — never by curation. The entire curated-artifacts system (this whole Artifacts-mode arc) exists specifically to fix that unfiltered-picker problem, and it's still not actually solved — wiring the param picker to real curation data is the eventual finish line this write-path consolidation is in service of, not a separate later nicety.

    - **PR 5 — Metadata/associations, editable**: wires the Edit-button flow above to the unified metadata PATCH (PR 4c). Deliberately separate from 4b — how editing should feel is worth answering with the read-only version already in hand, not guessed at alongside it. Not started; UX settled while looking at the built 4b panel, briefly considered and reversed one alternative along the way, worth keeping:
      - **Explicit "Edit" button, reconfirmed — not implicit dirty-state-triggers-edit-mode.** Seeing the real 4b panel raised the alternative of skipping a mode toggle entirely (start editing just by changing a field; Save/Discard appear once dirty). Considered and rejected: the earlier reasoning for an explicit gate (`docs/UI_WORKSPACE_MILESTONE.md`'s pre-existing "leaning explicit Edit button... not ambient" call, made before 4b was built) still holds now that the panel is real — a stray click on a param checkbox while just browsing shouldn't silently start an edit. If the explicit button ends up feeling like unnecessary friction in practice, revisit then; not removing it speculatively now.
      - **Button placement: bottom-right of the metadata panel, one slot reused across modes** — an "Edit" button there when read-only, replaced by "Save"/"Cancel" in the same spot once editing. Same pattern Sims' authoring flow already uses for its own Save/Cancel row, not a new convention. Also deliberately keeps this action area scoped to the metadata panel specifically (not a shared/page-level toolbar) — answers a forward-looking concern for free: PR 6+'s content-authoring flow (a separate concern, likely living in the middle content panel) will need its own action area, and keeping this one visually contained to the metadata panel should keep "editing metadata" and "authoring content" from reading as the same mode.
      - **Wording: "Cancel," not "Discard,"** for the button that exits edit mode without saving. Different call than the navigation-guard modal below — there, a floating standalone dismiss button reading "Cancel" would be ambiguous (cancel the navigation? cancel the edits?). Paired directly next to "Save" inside the panel, "Cancel" is the standard, unambiguous edit-record pattern — no separate ambiguity to avoid.
      - **Navigating away with unsaved changes shows a blocking modal — but only scoped to selecting a different artifact within Artifacts mode**, the one case that already routes through Redux (`selectArtifact`) and so is actually interceptable. Informational only, no action buttons that save or discard from inside the modal (deliberately, so it can't be mistaken for a shortcut around the panel's own Save/Cancel) — just a message like "Save or discard metadata changes before viewing another artifact" and a single dismiss button. **Button copy settled as "Got it"** (not "Finish editing," considered and rejected — reads too close to "I'm done, save it," reintroducing the exact save/cancel ambiguity this modal exists to avoid) — reads quickly, non-technical, unambiguously just closes the modal and returns to the still-dirty panel.
      - **Save is still the only point an API call is made — and now literally one call, not several.** Superseded by PR 4c's consolidation below: Save fires exactly one PATCH to the unified metadata endpoint with the current draft as one `ArtifactMetadata` object, not a fan-out of `associateArtifact`+curate/uncurate calls. Nothing autosaves or fires on individual field changes either way.
      - **State architecture: `isEditing` and the in-progress draft live in the `flow-version-artifacts-slice` Redux slice, not local `useState` on the panel.** The draft-copy pattern (each Field's `value`/`onChange` reads from and writes into a draft object, not directly into the fetched artifact — same controlled-component shape `SwitchField`/`CuratedParamsField` were already built for in 4b) needs _some_ owner for that state, but local state is destroyed the moment `FlowVersionArtifactMetadataPanel` unmounts, which happens on any navigation away from Artifacts mode. The slice already owns `selectedArtifactHash` and is the natural home — putting the draft there too means it survives that unmount/remount: navigate to Run mode mid-edit, come back, and it's exactly where you left it.
      - **Deliberately not pursuing navigation-blocking beyond the one case above — investigated and rejected, not just deferred.** Looked at `react-router`'s `useBlocker` for blocking navigation anywhere in the app (not just artifact selection); confirmed directly against the installed source (`react-router@7.13.0`) that it requires a _data router_ (`createBrowserRouter`/`RouterProvider`), which `apps/web-app/src/main.tsx` doesn't use (plain `<BrowserRouter>`) — adopting it would mean migrating the app's router setup first, not a small addition. Decided against regardless of that cost: letting someone leave Artifacts mode mid-edit to check something in another mode and come back is an intentional workflow, not just an accident to guard against, and the Redux-slice fix above already means nothing is actually lost by allowing it — so blocking would only add friction to a sometimes-deliberate action, without buying any additional safety.
      - **Known, accepted gap: this doesn't solve "I thought I saved that."** Someone can leave with unsaved edits, not return to that artifact for a while, and forget they were mid-edit — the draft survives, but nothing currently surfaces that one exists. A small future signal (a badge/dot indicating unsaved metadata changes on an artifact) is a plausible fix, raised in discussion but deliberately not designed now — a detail for later, not this PR's problem to solve.

      - **The RTK Query mutation now exists (`useUpdateArtifactMetadataMutation`, added in PR 4c) but isn't wired into any UI yet — that's this PR's job.** `apps/web-app/src/redux/api/artifacts-api.ts` previously had zero mutation endpoints for any of this; PR 4c added exactly one (matching the one-endpoint consolidation, not the three originally assumed pre-consolidation). PR 5 is purely wiring: draft state, the Edit/Save/Cancel flow above, and calling this mutation on Save.

      - **Confirmed, not a decision: `InputField` needs an actual `onChange` prop added.** It's hardcoded `readOnly` today with no callback path at all — unlike `SwitchField`/`CuratedParamsField`, which were deliberately built edit-mode-ready in 4b. Known gap already, not new information; just needs doing.

      - **Resolved, not just accepted: the "Save composes multiple API calls" friction is fixed by PR 4c's consolidation, not papered over.** One unified `ArtifactMetadata` PATCH call replaces the originally-envisioned fan-out of `associateArtifact`+curate/uncurate calls — see PR 4c above for the full reasoning.

      - **No toast/error system existing is fine — deferred on purpose, not a PR 5 decision.** Same already-tracked gap as `docs/todo.md`'s existing note (from Sims mode's Save flow). PR 5 doesn't need to solve it, just live with whatever ad hoc treatment gets used meanwhile.

      - **Confirmed fine: 4b hiding curated-but-type-incompatible params.** Once the backend gap (`curateArtifactForParam` not enforcing `isArtifactCompatible`) is fixed as part of PR 4c's write path, this situation shouldn't be able to arise going forward in the current system — so no special edit-mode handling for it is needed.

      - **Cancel is instant — no confirm-before-discard step.**

- **Edit mode, actually editable** — currently just a read-only copy of View. Real scope unknown; likely a substantial lift given flow authoring/validation/publish-state semantics aren't designed yet (e.g. whether a published version becomes immutable and editing forks a new draft version).
- **A full-step "resolved preview" view — parked, not scoped.** Raised while building Step Results: a view showing _every_ field of a step (matching `StepHttpJsonDetails`' full field list — URL/Method/Headers/Body/Exports — not just fields with refs), with each field showing its bound param value before a run and its actual resolved value after. Two distinct gaps motivate this, not one:
  - Fields with zero refs are invisible in Run mode entirely — only Edit/View's `StepHttpJsonDetails` shows them, and only unresolved.
  - Fields that _do_ have refs are only ever shown at leaf-`bindPath` granularity in Field Resolution (e.g. just `body.messages[1].content`), never composed back up into the whole logical field (`body`, `headers`) as it was actually sent — if `body` has refs at two different nested paths, Field Resolution shows two disjoint leaf rows, never one composed `body` object with both substitutions sitting in their natural positions. The natural grain for "the whole field, fully resolved" is the same field-level granularity `StepHttpJsonDetails` already uses; Field Resolution deliberately went finer because "what did this one reference become" wants the leaf, not the whole object.

  Also explicitly not Run-History-specific, and connects to an existing feature, not just Field Resolution: the per-param "show usages" report (`findParamRefs`/`renderParamRefReport`, `FlowVersionRunParamRow`) already lets you click a _param_ and see a markdown report of everywhere it's used — useful, but it's the reverse direction (param → its usages, leaf-level, textual report). What's missing is the other direction: click a _step_ in the live Params panel and see that whole step's fields composed with the currently-selected param artifacts filled in, ahead of submitting — same underlying resolution logic (`resolveRefHash`/`buildRefUsage`/`foldResolvedField`), organized by step instead of by param, at whole-field instead of leaf granularity. Deliberately not built yet because it's unclear how much it'd repeat Field Resolution/References; the resolution reached in discussion: judge that by whether it answers a _different question_ (a holistic "what did this step look like as one document" snapshot vs. drill-down provenance), not by whether it reuses the same underlying resolved-value logic — reusing `foldResolvedField` etc. across views isn't repetition, showing the same field list with the same framing twice would be. Needs its own design pass before it's scope, the same way Step Results got one.

- **Replay — acknowledged valuable, not scheduled (not "no reason exists" anymore).** Watching a historical run's execution animate at some controlled pace (a play-head over the event timeline instead of a range selection, reconstructing state at any point rather than just the final one) — technically straightforward given the architecture (`deriveStepRunInfo` already works on any event array, live or historical; this is UI/timing work, not new plumbing). The concrete reason it's worth building, not just "it'd be cool": it lets you re-experience the feel of watching a run live — pacing, what happened when — which a final-state view structurally can't reproduce. Still not the first thing to build; no committed timing.

## Evals: deliberately its own separate, future milestone

Explicitly not part of this arc's near-term scope — this is a real data-model rearchitecture, not a UI tab, and deserves its own milestone the same way the original eval vertical slice did (Milestone #9). Current state: evals are embedded inside the flow definition itself (an eval is just a flow with a judge step) — a smaller move than originally wanted. The actual goal: evals as standalone, reusable entities, decoupled from any one flow — a fixed template (System Prompt + Question + Context + Answer) that can be constructed and run against different flows. Inside this workspace, an Evals mode/tab would let you construct an eval run (bind the flow's actual output to the eval template's expected inputs) — results visualization might end up needing its own separate page if it doesn't fit inside that tab. Not scoped further here; needs its own dedicated design pass.
