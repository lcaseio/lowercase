- Real credentials for external APIs (e.g. a paid LLM) aren't safe to use yet: a flow step's `headers`/`body` get fully resolved (including anything credential-shaped) before the `job.<capability>.started` event is emitted, so a real secret would flow straight into the replay log and the web app's event viewer ŌĆö already demonstrated with a placeholder value (`"Bearer local-key"`) showing up verbatim in a pasted event. One candidate fix: resolve a `secrets` ref scope at the **worker** level (not the tool level), handing the tool a fully hydrated payload directly while separately emitting a redacted event to the bus (secret refs left as literal placeholders, never resolved in anything logged/replayed) ŌĆö this would preserve an invariant that a tool's only required input is its `job.*.started` payload, no new tool dependency. **But that invariant itself is explicitly not settled and wants more research** ŌĆö see below, since whether tools even stay in-process with the worker is an open question that changes where this boundary needs to sit. Actual secret storage (keychain vs. an encrypted-at-rest table with an externally-held key) is a separate, smaller decision once the resolution boundary is settled. Ties into the worker/tool interaction refactor already flagged in `CLAUDE.md` ŌĆö same root cause (too much raw detail riding inside job events).

- `EvalResultProjectionSink` and `SqlRunProjectionSink` both react to the same `run.completed` event via `ObservabilityTap`, and there's a real race between them: `SqlRunProjectionSink.handle()` is synchronous and internally fire-and-forgets its own flush (`void this.#flushLoop(...)`), so it never actually signals when its SQL write for that event has landed ŌĆö `ObservabilityTap` dispatching sinks in order, or even awaiting `handle()`, doesn't help, since `handle()` returns long before the write finishes. `EvalResultProjectionSink` needs that write to exist (it reads the eval run's `judge.score` export via `RunQueryPort.getRunDetail` right after `run.completed`) and can lose the race, silently producing "no judge.score export ŌĆö nothing to store." - Planned direction for a non-technical-user distribution: an Electron build, not Tauri, alongside a docker-compose-style path for technical users who want to self-host. Decided, not just leaning: Tauri would mean either rewriting the backend in Rust (not realistic ŌĆö it's TypeScript/Fastify) or running it as an actual bound localhost port that the webview hits over `fetch`, which is real attack surface (DNS-rebinding-style attacks against localhost servers are a documented risk class, not theoretical) ŌĆö mitigable with a shared-secret key the client must send, but Electron's IPC is process-scoped and avoids the exposure altogether without needing that mitigation. Electron is heavier, but there's already real precedent for the frontendŌåÆIPCŌåÆbackend-services shape in this exact codebase (`apps/desktop`, currently marked stale/dead ŌĆö this direction likely means rebuilding it rather than reviving it as-is, a call for whenever this actually gets built, not now).
  - Concrete mechanism for **not rewriting the backend**: keep every `apps/web-app` RTK Query endpoint exactly as-is (`query: () => ({ url, method, body })`) and swap only the `baseQuery` ŌĆö `fetchBaseQuery` for web/docker, and for Electron a custom `baseQuery` that hands that same `{url, method, body}` to `ipcRenderer.invoke(...)`, forwarded by the main process straight into **`fastify.inject({ method, url, payload })`** against the same app `apps/http-server/src/build-server.ts` already builds (separately from `apps/http-server/src/main.ts`, which is what actually calls `.listen()` ŌĆö the build/listen split Electron needs already exists, just needs exporting for cross-app use). `.inject()` never binds a socket or port ŌĆö it's not "running an HTTP server inside Electron" in the sense that matters for the Tauri comparison above; there's nothing for another process or web page to reach. No parallel IPC-to-service routing table to hand-maintain; Electron's IPC handler can likely be a thin, uniform "inject whatever request comes over this channel" passthrough.
  - **Verified gotcha, not just a guess**: this `.inject()` passthrough only works for normal request/response routes. Checked `light-my-request`'s actual source (`lib/response.js`) ŌĆö the injected callback fires on the response's `'finish'` event, i.e. only after `res.end()` is called, even with `payloadAsStream: true`. A long-lived SSE handler that holds the connection open and writes chunks over time without ending until `run.completed` would just hang the injected call forever. So the planned per-run SSE route (above) must NOT be reused via inject for the Electron case ŌĆö the main process should instead subscribe to the event bus directly (the same underlying primitive the SSE route itself is built on) and push events to the renderer via Electron's own native `webContents.send`, bypassing the injected-HTTP-route layer entirely for this one case.
  - That "subscribe to the bus directly" is safe specifically because the bus is a port (`EventBusPort`), not a concrete thing ŌĆö the same `bus.subscribe(...)` call works unchanged whether the wired adapter is in-process (what a self-contained Electron distribution would use ŌĆö no external infra, that's the whole point of Electron here) or, hypothetically, external infra (Redis/NATS/etc., relevant only to `apps/http-server`'s own possible future scaling, not to Electron). One real cost nuance if the bus is ever externally-backed, not a concern for Electron specifically: a fresh `bus.subscribe(...)` per live connection (one per SSE request, one per Electron IPC watch-channel) is effectively free against an in-process EventEmitter but not necessarily free against a real pub/sub broker at scale ŌĆö worth remembering if the SSE route's pattern ever gets reused in a deployment where the bus isn't in-process.
  - Side effect worth knowing about later: this would have Electron share the exact same wiring (`createServices()`, the full Fastify app) as `apps/http-server`/`apps/cli`, rather than the separate, narrower `createRuntime()`/`WorkflowRuntime` path `apps/desktop` uses today (see `CLAUDE.md`'s note on the two incomplete wiring paths) ŌĆö a real step toward that eventual consolidation, not just an Electron concern. Don't do the bigger move of extracting the app-builder into `packages/runtime` now, though ŌĆö that area is explicitly flagged as unsettled; a temporary `apps/desktop` ŌåÆ `apps/http-server` dependency is a fine stopgap, with this Electron use case becoming one more concrete data point for how that consolidation should shape up later.
  - Directly raises the priority of the existing "real credentials aren't safe yet" item above: if the pitch is "a non-technical person pastes their own OpenAI key in," a resolved secrets-redaction boundary stops being a someday backlog item and becomes a blocker for this specific product plan ŌĆö even a user's own key logged verbatim into their own local replay/event history is a real leak vector (screen shares, support requests, accidental exports).
