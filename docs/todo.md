- CAS hashing conflates content identity with metadata: `FlowDefinition`'s full JSON (including fields like `description`, now `kind`) feeds `definitionHash`, so a purely descriptive edit forces a new `FlowVersion` even though nothing behaviorally changed. Known and accepted for a while, not urgent — `FlowVersion`'s `sequence`-based versioning already exists in SQL but isn't exercised by any real reuse/dedup logic yet. If tackled: likely split into a separate behavior-only `contentHash` (steps/start/params/outputs) alongside the existing full-content `definitionHash`, so two versions can share a `contentHash` while differing only in metadata.
- Real credentials for external APIs (e.g. a paid LLM) aren't safe to use yet: a flow step's `headers`/`body` get fully resolved (including anything credential-shaped) before the `job.<capability>.started` event is emitted, so a real secret would flow straight into the replay log and the web app's event viewer — already demonstrated with a placeholder value (`"Bearer local-key"`) showing up verbatim in a pasted event. One candidate fix: resolve a `secrets` ref scope at the **worker** level (not the tool level), handing the tool a fully hydrated payload directly while separately emitting a redacted event to the bus (secret refs left as literal placeholders, never resolved in anything logged/replayed) — this would preserve an invariant that a tool's only required input is its `job.*.started` payload, no new tool dependency. **But that invariant itself is explicitly not settled and wants more research** — see below, since whether tools even stay in-process with the worker is an open question that changes where this boundary needs to sit. Actual secret storage (keychain vs. an encrypted-at-rest table with an externally-held key) is a separate, smaller decision once the resolution boundary is settled. Ties into the worker/tool interaction refactor already flagged in `CLAUDE.md` — same root cause (too much raw detail riding inside job events).
- The capability/tool-registry extensibility vision (a tool declares a capability like `httpjson`, a registry tracks what's available, a step's `tool` field selects an implementation) is real and partially built (`StepCapCommonFields.tool?`, `ToolRegistry`, `allToolBindingsMap`) — but never proven out by a second real capability beyond `httpjson`/`mcp`. **Open research direction, genuinely undecided**: tools may not need to be in-process with the worker at all — a "tool" could be an out-of-process HTTP service the worker calls, with a possible SDK package (event-parsing helpers, response-shape helpers) letting a third party build a tool as a standalone service without forking or extending this repo's code at all. Would change where both the worker/tool payload boundary and the secrets-safety boundary (above) actually need to sit — the "tool only needs `job.*.started`" framing was useful for reasoning through the secrets question, but isn't a conclusion, just one candidate shape among several. Worker is acknowledged-bloated and wants new eyes before extending this further; likely the same refactor pass as the secrets item above.
- `EvalResultProjectionSink` and `SqlRunProjectionSink` both react to the same `run.completed` event via `ObservabilityTap`, and there's a real race between them: `SqlRunProjectionSink.handle()` is synchronous and internally fire-and-forgets its own flush (`void this.#flushLoop(...)`), so it never actually signals when its SQL write for that event has landed — `ObservabilityTap` dispatching sinks in order, or even awaiting `handle()`, doesn't help, since `handle()` returns long before the write finishes. `EvalResultProjectionSink` needs that write to exist (it reads the eval run's `judge.score` export via `RunQueryPort.getRunDetail` right after `run.completed`) and can lose the race, silently producing "no judge.score export — nothing to store." Band-aided for now with a short retry-with-backoff loop inside `EvalResultProjectionSink` (`#findScoreExportWithRetry`, `packages/observability/src/sinks/eval-result-projection.sink.ts`) rather than fixing the ordering at the source, since `SqlRunProjectionSink`'s dirty/flushing coalescing mechanism is already known to want a bigger refactor (carries over `run-history`'s in-memory index pattern) — a real fix means giving that sink a way to expose/await its actual flush completion, which touches that same pending rework. Revisit the retry band-aid once that refactor happens.

- Planned direction, not yet done: replace the observability websocket (`apps/http-server/src/routes/ws-route.ts`, `packages/services/src/ws.service.ts`) with per-run Server-Sent Events (`GET /api/runs/:runId/stream`, real `text/event-stream`, not a bespoke protocol) for Run mode's live view. Concrete motivation, not just simplification for its own sake: today's WS session handling is genuinely broken, not just "light" — `sockets` is a single global `Map` keyed by the literal string `"client"`, so a second connected browser tab silently steals live-run monitoring from the first (`request.ts`'s `sockets.get("client")` at run-start time). Per-run SSE connections fix this for free — each request is its own connection scoped to one runId, nothing shared to collide on — rather than needing bespoke per-client session ids bolted onto the existing WS. The original broader vision for the WS (a system-wide "what's running right now" dashboard, not scoped to one run) never got built and isn't being solved by this migration; if/when that view happens, it likely wants its own separate "active runs" firehose endpoint built on the same underlying "subscribe to the bus, stream to the response" server-side primitive, not a shared connection also used for per-run watching. Expected to be low-disruption on the frontend: the live-events pipeline already bypasses RTK Query's subscription model (a custom `ws` redux middleware dispatching into `events-slice`, not `onCacheEntryAdded`), so swapping the wire source to `EventSource.onmessage` shouldn't require touching that boundary.
- Planned direction for a non-technical-user distribution: an Electron build, not Tauri, alongside a docker-compose-style path for technical users who want to self-host. Decided, not just leaning: Tauri would mean either rewriting the backend in Rust (not realistic — it's TypeScript/Fastify) or running it as an actual bound localhost port that the webview hits over `fetch`, which is real attack surface (DNS-rebinding-style attacks against localhost servers are a documented risk class, not theoretical) — mitigable with a shared-secret key the client must send, but Electron's IPC is process-scoped and avoids the exposure altogether without needing that mitigation. Electron is heavier, but there's already real precedent for the frontend→IPC→backend-services shape in this exact codebase (`apps/desktop`, currently marked stale/dead — this direction likely means rebuilding it rather than reviving it as-is, a call for whenever this actually gets built, not now).
  - Concrete mechanism for **not rewriting the backend**: keep every `apps/web-app` RTK Query endpoint exactly as-is (`query: () => ({ url, method, body })`) and swap only the `baseQuery` — `fetchBaseQuery` for web/docker, and for Electron a custom `baseQuery` that hands that same `{url, method, body}` to `ipcRenderer.invoke(...)`, forwarded by the main process straight into **`fastify.inject({ method, url, payload })`** against the same app `apps/http-server/src/build-server.ts` already builds (separately from `apps/http-server/src/main.ts`, which is what actually calls `.listen()` — the build/listen split Electron needs already exists, just needs exporting for cross-app use). `.inject()` never binds a socket or port — it's not "running an HTTP server inside Electron" in the sense that matters for the Tauri comparison above; there's nothing for another process or web page to reach. No parallel IPC-to-service routing table to hand-maintain; Electron's IPC handler can likely be a thin, uniform "inject whatever request comes over this channel" passthrough.
  - **Verified gotcha, not just a guess**: this `.inject()` passthrough only works for normal request/response routes. Checked `light-my-request`'s actual source (`lib/response.js`) — the injected callback fires on the response's `'finish'` event, i.e. only after `res.end()` is called, even with `payloadAsStream: true`. A long-lived SSE handler that holds the connection open and writes chunks over time without ending until `run.completed` would just hang the injected call forever. So the planned per-run SSE route (above) must NOT be reused via inject for the Electron case — the main process should instead subscribe to the event bus directly (the same underlying primitive the SSE route itself is built on) and push events to the renderer via Electron's own native `webContents.send`, bypassing the injected-HTTP-route layer entirely for this one case.
  - That "subscribe to the bus directly" is safe specifically because the bus is a port (`EventBusPort`), not a concrete thing — the same `bus.subscribe(...)` call works unchanged whether the wired adapter is in-process (what a self-contained Electron distribution would use — no external infra, that's the whole point of Electron here) or, hypothetically, external infra (Redis/NATS/etc., relevant only to `apps/http-server`'s own possible future scaling, not to Electron). One real cost nuance if the bus is ever externally-backed, not a concern for Electron specifically: a fresh `bus.subscribe(...)` per live connection (one per SSE request, one per Electron IPC watch-channel) is effectively free against an in-process EventEmitter but not necessarily free against a real pub/sub broker at scale — worth remembering if the SSE route's pattern ever gets reused in a deployment where the bus isn't in-process.
  - Side effect worth knowing about later: this would have Electron share the exact same wiring (`createServices()`, the full Fastify app) as `apps/http-server`/`apps/cli`, rather than the separate, narrower `createRuntime()`/`WorkflowRuntime` path `apps/desktop` uses today (see `CLAUDE.md`'s note on the two incomplete wiring paths) — a real step toward that eventual consolidation, not just an Electron concern. Don't do the bigger move of extracting the app-builder into `packages/runtime` now, though — that area is explicitly flagged as unsettled; a temporary `apps/desktop` → `apps/http-server` dependency is a fine stopgap, with this Electron use case becoming one more concrete data point for how that consolidation should shape up later.
  - Directly raises the priority of the existing "real credentials aren't safe yet" item above: if the pitch is "a non-technical person pastes their own OpenAI key in," a resolved secrets-redaction boundary stops being a someday backlog item and becomes a blocker for this specific product plan — even a user's own key logged verbatim into their own local replay/event history is a real leak vector (screen shares, support requests, accidental exports).
