- `EvalResultProjectionSink` and `SqlRunProjectionSink` both react to the same `run.completed` event via `ObservabilityTap`, and there's a real race between them: `SqlRunProjectionSink.handle()` is synchronous and internally fire-and-forgets its own flush (`void this.#flushLoop(...)`), so it never actually signals when its SQL write for that event has landed — `ObservabilityTap` dispatching sinks in order, or even awaiting `handle()`, doesn't help, since `handle()` returns long before the write finishes. `EvalResultProjectionSink` needs that write to exist (it reads the eval run's `judge.score` export via `RunQueryPort.getRunDetail` right after `run.completed`) and can lose the race, silently producing "no judge.score export — nothing to store." - Planned direction, not yet done: replace the observability websocket (`apps/http-server/src/routes/ws-route.ts`, `packages/services/src/ws.service.ts`) with per-run Server-Sent Events (`GET /api/runs/:runId/stream`, real `text/event-stream`, not a bespoke protocol) for Run mode's live view. Concrete motivation, not just simplification for its own sake: today's WS session handling is genuinely broken, not just "light" — `sockets` is a single global `Map` keyed by the literal string `"client"`, so a second connected browser tab silently steals live-run monitoring from the first (`request.ts`'s `sockets.get("client")` at run-start time). Per-run SSE connections fix this for free — each request is its own connection scoped to one runId, nothing shared to collide on — rather than needing bespoke per-client session ids bolted onto the existing WS. The original broader vision for the WS (a system-wide "what's running right now" dashboard, not scoped to one run) never got built and isn't being solved by this migration; if/when that view happens, it likely wants its own separate "active runs" firehose endpoint built on the same underlying "subscribe to the bus, stream to the response" server-side primitive, not a shared connection also used for per-run watching. Expected to be low-disruption on the frontend: the live-events pipeline already bypasses RTK Query's subscription model (a custom `ws` redux middleware dispatching into `events-slice`, not `onCacheEntryAdded`), so swapping the wire source to `EventSource.onmessage` shouldn't require touching that boundary.
- Planned direction for a non-technical-user distribution: an Electron build, not Tauri, alongside a docker-compose-style path for technical users who want to self-host. Decided, not just leaning: Tauri would mean either rewriting the backend in Rust (not realistic — it's TypeScript/Fastify) or running it as an actual bound localhost port that the webview hits over `fetch`, which is real attack surface (DNS-rebinding-style attacks against localhost servers are a documented risk class, not theoretical) — mitigable with a shared-secret key the client must send, but Electron's IPC is process-scoped and avoids the exposure altogether without needing that mitigation. Electron is heavier, but there's already real precedent for the frontend→IPC→backend-services shape in this exact codebase (`apps/desktop`, currently marked stale/dead — this direction likely means rebuilding it rather than reviving it as-is, a call for whenever this actually gets built, not now).
  - Concrete mechanism for **not rewriting the backend**: keep every `apps/web-app` RTK Query endpoint exactly as-is (`query: () => ({ url, method, body })`) and swap only the `baseQuery` — `fetchBaseQuery` for web/docker, and for Electron a custom `baseQuery` that hands that same `{url, method, body}` to `ipcRenderer.invoke(...)`, forwarded by the main process straight into **`fastify.inject({ method, url, payload })`** against the same app `apps/http-server/src/build-server.ts` already builds (separately from `apps/http-server/src/main.ts`, which is what actually calls `.listen()` — the build/listen split Electron needs already exists, just needs exporting for cross-app use). `.inject()` never binds a socket or port — it's not "running an HTTP server inside Electron" in the sense that matters for the Tauri comparison above; there's nothing for another process or web page to reach. No parallel IPC-to-service routing table to hand-maintain; Electron's IPC handler can likely be a thin, uniform "inject whatever request comes over this channel" passthrough.
  - **Verified gotcha, not just a guess**: this `.inject()` passthrough only works for normal request/response routes. Checked `light-my-request`'s actual source (`lib/response.js`) — the injected callback fires on the response's `'finish'` event, i.e. only after `res.end()` is called, even with `payloadAsStream: true`. A long-lived SSE handler that holds the connection open and writes chunks over time without ending until `run.completed` would just hang the injected call forever. So the planned per-run SSE route (above) must NOT be reused via inject for the Electron case — the main process should instead subscribe to the event bus directly (the same underlying primitive the SSE route itself is built on) and push events to the renderer via Electron's own native `webContents.send`, bypassing the injected-HTTP-route layer entirely for this one case.
  - That "subscribe to the bus directly" is safe specifically because the bus is a port (`EventBusPort`), not a concrete thing — the same `bus.subscribe(...)` call works unchanged whether the wired adapter is in-process (what a self-contained Electron distribution would use — no external infra, that's the whole point of Electron here) or, hypothetically, external infra (Redis/NATS/etc., relevant only to `apps/http-server`'s own possible future scaling, not to Electron). One real cost nuance if the bus is ever externally-backed, not a concern for Electron specifically: a fresh `bus.subscribe(...)` per live connection (one per SSE request, one per Electron IPC watch-channel) is effectively free against an in-process EventEmitter but not necessarily free against a real pub/sub broker at scale — worth remembering if the SSE route's pattern ever gets reused in a deployment where the bus isn't in-process.
  - Side effect worth knowing about later: this would have Electron share the exact same wiring (`createServices()`, the full Fastify app) as `apps/http-server`/`apps/cli`, rather than the separate, narrower `createRuntime()`/`WorkflowRuntime` path `apps/desktop` uses today (see `CLAUDE.md`'s note on the two incomplete wiring paths) — a real step toward that eventual consolidation, not just an Electron concern. Don't do the bigger move of extracting the app-builder into `packages/runtime` now, though — that area is explicitly flagged as unsettled; a temporary `apps/desktop` → `apps/http-server` dependency is a fine stopgap, with this Electron use case becoming one more concrete data point for how that consolidation should shape up later.
  - Directly raises the priority of the existing "real credentials aren't safe yet" item above: if the pitch is "a non-technical person pastes their own OpenAI key in," a resolved secrets-redaction boundary stops being a someday backlog item and becomes a blocker for this specific product plan — even a user's own key logged verbatim into their own local replay/event history is a real leak vector (screen shares, support requests, accidental exports).

  - **`data.status` field inside those events** (`"success"`/`"failure"`) — present tense, deliberately. Confirmed earlier this session (via direct code trace) that this is currently 100% redundant with the event type in every real emission path — `StepCompletedData.status` is even typed as the literal `"success"`, so the type system forbids divergence today. - **New idea, genuinely undecided, not started: artifact "edit" and deletion, and a draft/publish delineation that would make it tractable.** Raised while designing the unified `ArtifactMetadata` write path for Artifacts mode PR 5 — a separate, harder question surfaced alongside it, deliberately not pulled into that PR's scope.
  - **"Editing" an artifact's content isn't a real operation given CAS semantics — it's always create-new, sometimes-plus-delete-old.** Content is hash-addressed (`hash = sha256(content)`); there's no such thing as mutating content at a fixed hash without it becoming a different artifact. A user's "I made a mistake, let me fix it" experience is really: create a new artifact, optionally re-point whatever referenced the old one, optionally garbage-collect the old one if nothing else needs it. Nothing about `ArtifactMetadata` (SQL-side, mutable-by-design metadata) needs to anticipate this — metadata and content are already cleanly separate concerns.

  - **Two different levels of "safe to change," worth keeping distinct.** Metadata-level associations (curations, `flowId`/`flowVersionId` association, `label`) are comparatively low-stakes — changing or removing one doesn't corrupt anything historical. What's actually load-bearing and must never dangle is a hash that's been _consumed or produced_ by a run/event (run params, step outputs, exports) — that's the real ledger; the traceability guarantee lives there, not in the curation/association layer.

  - **Deletion needs a reachability check — the real design question is eager-vs-lazy, not whether to check at all.** Considered and leaning away from a maintained boolean/flag set whenever a run/sim starts using an artifact — that's a write on every reference-creating action (run/sim creation, frequent) to speed up a check that's only ever needed at delete time (rare), the wrong end of that tradeoff. Also a real drift risk: a maintained flag can silently fall out of sync if any code path forgets to set it — the same class of problem already hit once in this codebase (`SqlRunProjectionSink`'s carried-over-from-filesystem-era projection mechanism, flagged elsewhere in this file). Leaning instead toward a lazy, on-demand reachability query at delete-time across every real reference source (run params, step outputs/exports, curated-param associations, the artifact's own `flowId`/`flowVersionId`) — correct against ground truth, no drift possible, cost only paid when actually needed. Exhaustiveness is the hard part either way: missing even one reference source turns "delete" into silent corruption elsewhere.

  - **A draft/publish delineation would make this dramatically more tractable, not just nicer UX.** If an artifact starts as a "draft" — not eligible for any consuming action (can't be a run param, can't be curated, nothing that touches runs/events can reference it) — then by construction a draft can _never_ be referenced, so it can be freely edited/deleted with zero reachability risk, no check needed at all. Only once "published" does an artifact become consumable, and only published artifacts ever need the expensive reachability check before deletion. This narrows the hard problem (is this referenced _anywhere_) down to a much smaller one (is this published) — the same well-trodden pattern as draft-vs-published content in a CMS, not a novel risky idea.

  - Same underlying concern as the existing "deleting a sim needs referential-integrity rules" item elsewhere in this file — one more concrete instance of "this data model is intentionally interlocked to preserve traceability," not a separate problem. Not designed, not scoped, explicitly not part of PR 5.
